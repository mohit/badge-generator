Vision

The Let's Encrypt moment for credentials

Current state

Portable trust

Method

Open trust rails first

What's here now

Trust model

Three-state verifier output

DOMAIN_VERIFIED_SIGNATURE, DEMO_DOMAIN_VERIFIED_SIGNATURE, or UNVERIFIED, with explicit caveat about assessment quality.

Security posture

Public verifier hardening

SSRF protections, request limits, trust-write rate limits, and cooldown controls for abuse resistance.

Interfaces

API, CLI, MCP, llms.txt

Human and agent workflows share the same trust primitives and verification semantics.

How issuer identity works

Canonical issuer identity

Verifier identity is domain-first: issued by <domain>.

Claimed names are secondary

Issuer profile names are displayed as claims, not canonical trust anchors.

TLS org/CN is metadata only

Certificate org fields may describe a hosting provider. Domain ownership remains the trust anchor.

Now, next, later

Now: Domain/key trust

Public verification, trust discovery APIs, and prompt-to-badge activation for early adopters.

Next: Issuer onboarding UX

Guided flows for non-technical institutions: key generation, well-known hosting, and first issued badge.

Later: Circles of trust

Vouching networks for higher-order reputation layered on top of domain-bound signature trust.
